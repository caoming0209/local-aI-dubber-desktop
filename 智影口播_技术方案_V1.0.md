# 一、产品定位

智影口播是一款本地全离线运行的 AI 数字人口播视频生成工具。

目标：

> 单张图片 + 文案 + 本地零样本声音克隆→ 自动生成 1080P 30fps 数字人口播视频→ 在 8GB 显存 NVIDIA 显卡上稳定运行→ 支持批量生产（建议 ≤30 条）
> 
> 

核心原则：

- 全离线运行

- 严格串行

- 显存可控

- 自动降级保护

- 稳定优先于极限效果

# 五、完整生成流程

## 5.1 阶段1 文案处理（CPU）

- 自动断句

- 单段 ≤300 字（超出自动拆分，拆分后视频自动拼接，保障用户体验）

- 超长自动拆分（拆分逻辑按标点+语义结合，避免拆分导致口型断裂）

## 5.2 阶段2 声音克隆与合成

### 5.2.1 克隆流程

上传音频 → Whisper转文本 → 提取speaker embedding → 保存embedding.json

不生成独立模型副本（节省磁盘空间与内存占用）。

### 5.2.2 推理规范

- FP16

- batch_size=1

- inference_mode

- 推理完成立即释放显存（显存峰值 ≤3GB，符合Memory Guard监控标准）

## 5.3 阶段3 单张图片驱动视频

- 强制 resize 512x512（8GB显存稳定运行核心前提）

- 内部生成512分辨率（后续通过FFmpeg拉伸至目标输出分辨率）

- 输出 base_video.mp4

- 不生成3D模型（降低显存与内存消耗）

显存峰值 ≤5.5GB（预留800MB buffer，规避显存不足风险）

## 5.4 阶段4 口型精修

- 仅处理口周ROI（减少无效计算，降低显存占用）

- FP16（轻量模式可切换为低精度，进一步释放显存）

显存峰值 ≤5GB

## 5.5 阶段5 画质增强

- 分帧批处理（降低单帧显存占用）

- batch_size=5（32GB内存可提升至8，优化处理效率）

- tile=256

显存峰值 ≤4GB

## 5.6 阶段6 字幕对齐

- Whisper small

- 输出ASS字幕（后续可扩展SRT/VTT格式，满足多场景需求）

- 字幕对齐精度 ≤100毫秒（保障口型与字幕同步）

## 5.7 阶段7 合成输出

- FFmpeg 合成（CPU执行，不占用显存）

- H264 编码

- 输出 1080P 30fps（默认最优方案，可按需切换为720P）

# 六、显存管理机制（优化补充版）

新增模块：Memory Guard（核心保障8GB显存稳定运行，补充异常处理与闭环逻辑）

阶段执行前必须检测：free_vram ≥ 阶段需求 + 800MB buffer（buffer可根据系统显存波动动态调整，避免驱动/系统占用导致OOM）

## 6.1 安全阈值

|阶段|required_mb|
|---|---|
|TTS|2500|
|SadTalker|5500|
|MuseTalk|5000|
|Enhance|4000|
## 6.2 补充优化

- 显存检测：采用CUDA 11.8适配的nvmlDeviceGetMemoryInfo API，精准获取free_vram，排除系统预留、其他进程占用的显存，避免误判

- 显存清理：阶段执行完毕、单条任务完成后，自动执行torch.cuda.empty_cache()清理显存，批量任务每10条重启模型进程，彻底释放显存碎片

- 异常处理：若检测到显存不足，先触发轻量模式；若触发OOM异常，立即中断当前任务，记录错误日志，提示用户调整分辨率或关闭增强模块，避免程序崩溃

# 七、轻量模式（优化补充版）

## 7.1 触发条件

- 显存不足（free_vram < 阶段需求 + 800MB buffer）

- OOM异常（显存溢出触发）

## 7.2 降级规则

|模块|降级策略|
|---|---|
|SadTalker|固定512分辨率（不可调整，保障基础运行）|
|MuseTalk|低精度（INT8），进一步降低显存占用|
|GFPGAN|关闭（人脸修复功能停用，释放约1GB显存）|
|超分|关闭（超分功能停用，释放约0.8GB显存）|
## 7.3 补充优化

- 恢复逻辑：轻量模式触发后，实时监控显存状态，若后续任务显存充足（free_vram ≥ 阶段需求 + 1GB buffer），自动切回正常模式

- 效果兜底：多模块同时降级时，输出视频添加“轻量模式”水印提示，保障用户感知，避免误以为是功能异常

# 八、批量运行策略（优化补充版）

- 单批建议 ≤30 条（量化依据：30条任务显存碎片累积、磁盘IO压力、内存占用均在8GB显存+32GB内存的安全范围内）

- 严格串行（规避多任务并发导致的显存溢出）

- 每条完成后清理显存（执行torch.cuda.empty_cache()，避免显存碎片累积）

- 每10条建议重启模型进程（重启耗时约30秒，重启后自动加载模型权重、恢复任务上下文，彻底释放显存碎片）

## 8.1 补充优化

- 断点续传：批量运行中程序崩溃，重启后可从失败的那条任务继续执行，无需从头重跑，提升批量处理效率

- 任务监控：批量运行时，实时展示每条任务的进度、显存占用、耗时，异常任务标注并提示原因

# 九、交互调整（符合技术约束，补充优化）

## 9.1 删除功能

- 60fps 插帧（8GB显存无法支撑，避免显存溢出；可作为高级模式，仅对12GB+显卡开放，后续可扩展）

- 实时增强预览（占用显存过高，易导致OOM；替换为“离线预览”，生成临时低分辨率预览图，不占用大量显存）

- 并行生成（8GB显存无法支撑多任务并行，严格串行更稳定）

## 9.2 新增交互提示

- 显存实时状态展示（采样频率1秒/次，显示已用/可用/总显存，接近阈值时红色预警）

- 当前模式：正常 / 轻量（模式切换时给出提示，轻量模式标注已关闭的模块）

- 批量安全上限提示（推荐30条，超出提示“超出安全上限，可能导致显存不足或程序崩溃”）

- 单段文本长度提示（超出300字自动拆分，提示拆分条数、拆分逻辑）

- 临时文件清理：任务完成后，自动删除base_video.mp4、分帧图片等临时文件，节省磁盘空间

# 十一、性能预期（8GB显卡，补充量化优化）

- 单条视频：2.5~3分钟（拆分至各阶段：文案处理≈10秒、TTS≈30秒、SadTalker≈60秒、MuseTalk≈40秒、画质增强≈20秒、字幕对齐≈10秒、合成≈10秒）

- 批量30条：约90分钟（包含每10条重启模型进程的耗时，单条平均3分钟）

## 11.1 补充说明

性能测试基准：测试显卡为NVIDIA RTX 3050（8GB）、CPU i5-12400、32GB RAM、1TB SSD，单条视频文本长度150字、图片分辨率1080P，正常模式下测试得出；轻量模式耗时增加30%~50%，768×768原生分辨率耗时增加约40%。

# 十二、最低硬件要求（补充兼容边界，优化明确性）

- NVIDIA GPU ≥ 8GB（支持Ampere/Turing架构，CUDA compute capability≥7.0，确保模型兼容）

- RAM ≥16GB（32GB最优，可优化批量处理效率；16GB包含模型加载+视频帧处理+系统占用，无冗余）

- SSD ≥100GB（包含模型文件≈30GB、临时视频文件≈20GB、输出文件≈40GB、系统预留≈10GB，保障磁盘IO流畅）

- CUDA 驱动 ≥520.61.05（适配CUDA 11.8，避免兼容性问题；提示“建议更新至最新驱动，提升运行稳定性”）

## 12.1 补充说明

- 非NVIDIA显卡（AMD/Intel）：暂不支持GPU加速，仅可CPU运行，生成速度大幅降低（单条视频耗时≈10分钟），不推荐使用

- 资源瓶颈提示：CPU低于i5-10400、RAM低于16GB、SSD低于100GB时，启动时提示“硬件配置低于最低要求，可能导致运行卡顿、崩溃”

# 十三、分辨率适配与硬件能力补充说明（核心建议植入）

## 13.1 核心分辨率支持逻辑（8GB显存+32GB内存）

结合8GB显存的核心瓶颈的32GB内存的冗余优势，明确各分辨率适配方案，兼顾稳定性、效果与速度，具体如下：

|目标输出分辨率|核心生成分辨率|可行性|显存/内存支撑依据|额外代价/优化措施|
|---|---|---|---|---|
|720P|512×512（拉伸）|极高|8GB显存峰值≤5.5GB，32GB内存无压力，磁盘IO负载低|无额外代价，拉伸幅度更小，画面更清晰，耗时略低于1080P|
|1080P|512×512（拉伸）|高|8GB显存满足SadTalker 512分辨率需求，32GB内存支撑FFmpeg拉伸数据处理，无内存瓶颈|文档默认最优方案，单条耗时2.5~3分钟，全程稳定，兼顾效果与速度|
|768×768|768×768（原生）|中|8GB显存峰值≈8.5GB（接近阈值），32GB内存完全支撑帧缓存，无内存压力|需触发轻量模式（关闭GFPGAN/超分），耗时增加30%~50%，偶发显存波动需重试，画面细节略有提升|
|2K/4K|-|极低|8GB显存无法支撑1024×1024以上原生生成分辨率，即使内存充足也无法突破显存瓶颈|完全不可行，启动时提示“当前硬件无法支撑2K/4K分辨率，建议切换至1080P或720P”|
## 13.2 关键技术补充

- 分辨率拉伸逻辑：512×512原生生成后通过FFmpeg拉伸至1080P/720P，是8GB显存下“低显存占用+高输出分辨率”的折中方案，数字人口型/表情核心细节由512×512生成链路决定，拉伸仅提升画面尺寸，无核心效果损失，且拉伸过程由CPU执行，不占用显存。

- 内存冗余价值：32GB内存（远超16GB最低要求）可优化批量处理效率（如批量30条时无需频繁清理帧缓存）、提升分帧批处理的batch_size（如画质增强阶段batch_size可从5提升至8），减少磁盘IO交互，但无法突破显存对“原生生成分辨率”的限制（显存是分辨率提升的核心瓶颈）。

- 分辨率与耗时关系：原生分辨率从512×512提升至768×768，单条耗时增加约40%（批量30条从90分钟增至126分钟），且需关闭人脸修复/超分模块，画面细节略有损失；目标分辨率从1080P降至720P，单条耗时减少约10%，稳定性进一步提升。

## 13.3 推荐配置策略

- 最优选择：优先选择1080P输出（512×512核心生成），兼顾效果、速度与稳定性，适配绝大多数8GB显存用户需求。

- 极限尝试：仅在特殊场景（对画面细节要求极高，可接受耗时增加）下尝试768×768原生分辨率，需提前开启轻量模式，做好重试准备。

- 不可行场景：2K/4K分辨率完全超出8GB显存承载能力，不建议尝试，避免程序崩溃、显存溢出等问题。

# 十四、资源监控与告警（新增，补充核心优化建议）

为保障8GB显存下稳定运行，新增资源监控与告警模块，覆盖显存、CPU、内存、磁盘IO全维度，便于问题排查与异常处置。

## 14.1 监控指标

- 显存：已用/可用/总显存，采样频率1秒/次，记录各阶段峰值

- CPU：占用率、核心使用率，采样频率2秒/次，阈值≥90%告警

- 内存：已用/可用/总内存，采样频率2秒/次，阈值≥85%告警

- 磁盘IO：读写速度、磁盘剩余空间，采样频率5秒/次，剩余空间≤10GB告警

## 14.2 日志规范

日志（job.log）需包含以下字段：时间戳、任务ID、阶段名称、耗时、资源占用（显存/CPU/内存）、执行状态（成功/失败）、异常信息（若失败），日志格式统一，便于后续统计与分析。

## 14.3 异常告警

- 显存接近阈值（≥7.2GB）：前端红色预警，提示“显存不足，即将触发轻量模式”

- 磁盘空间不足（≤10GB）：前端弹窗提示，暂停当前任务，建议清理磁盘空间

- CPU/内存过载：前端提示“资源占用过高，任务可能卡顿”，自动降低当前阶段batch_size

- 任务失败：标注失败原因（如OOM、磁盘IO错误），提示解决方案（如关闭增强模块、清理磁盘）
